📖 Glossary & Reference 53-term DWH/SSIS/SSAS/Power BI Glossary · Star vs Snowflake Framework · Table Role Classification Guide
📚 Table A — DWH Terms Glossary (53 terms)
#TermCategoryDefinition
1GrainSchema DesignThe finest level of detail captured in a fact table. Defines what one row represents.
2Surrogate Key (SK)Schema DesignA system-generated integer key assigned in the DWH, independent of the source system. Enables SCD management.
3Natural Key (NK)Schema DesignThe business identifier from the source system (e.g., CustomerID). Used for lookup, not as the DWH primary key.
4SCD Type 0SCDFixed value — dimension attribute never changes. Historical value is retained forever.
5SCD Type 1SCDOverwrite — dimension attribute is updated in place. No history is maintained.
6SCD Type 2SCDFull history — a new row is added for each change. Is_Current, StartDate, EndDate track the timeline.
7SCD Type 3SCDPrior value — adds a column to store the previous value. Only one historical value retained.
8SCD Type 6SCDHybrid of Type 1+2+3. Overwrites current value, adds new row, and maintains a current-value column on all rows.
9Conformed DimensionSchema DesignA dimension shared identically across multiple fact tables, enabling drill-across queries.
10Degenerate DimensionSchema DesignA dimension key with no corresponding dimension table (e.g., InvoiceNumber stored directly in the fact table).
11Bridge TableSchema DesignA table used to resolve Many-to-Many relationships in a star schema, often with weighting factors.
12Fact TableSchema DesignCentral table in a star schema containing measurable business events and foreign keys to dimensions.
13Dimension TableSchema DesignDescriptive table in a star schema providing context for fact table measures (who, what, where, when, how).
14Star SchemaArchitectureDWH schema pattern with one central fact table surrounded by denormalised dimension tables.
15Snowflake SchemaArchitectureDWH schema pattern where dimension tables are normalised into sub-dimensions. More complex, less query-friendly.
16ETLArchitectureExtract, Transform, Load — the process of moving and transforming data from source to DWH.
17OLTPArchitectureOnline Transaction Processing — operational source system optimised for transactional reads/writes.
18OLAPArchitectureOnline Analytical Processing — analytical system optimised for complex queries over large datasets.
19Staging LayerArchitectureRaw copy of source data in the DWH database. First landing zone before any transformation.
20Unknown MemberSchema DesignA row with SK=-1 in each dimension representing 'not applicable' or 'unknown'. Prevents orphan keys in facts.
21IdempotencyETLProperty of an ETL process that produces the same result whether run once or multiple times — safe to re-run.
22KimballMethodologyBottom-up DWH methodology using dimensional modelling (star schemas) per business process, built by Ralph Kimball.
23InmonMethodologyTop-down DWH methodology using a centralised normalised enterprise model, built by Bill Inmon.
24Slowly Changing DimensionSCDA dimension where attribute values change slowly over time, requiring a strategy to handle historical tracking.
25Role-Playing DimensionSchema DesignA single dimension table used multiple times in the same fact table for different contexts (e.g., Order Date, Ship Date).
26Junk DimensionSchema DesignA single dimension table consolidating multiple low-cardinality flags/indicators to reduce fact table width.
27Aggregate FactSchema DesignA pre-summarised fact table storing higher-level aggregations for query performance.
28Accumulated SnapshotSchema DesignA fact table with one row per business process instance, updated as milestones are reached.
29Periodic SnapshotSchema DesignA fact table capturing the state of a business process at regular intervals (e.g., daily inventory levels).
30Transaction FactSchema DesignA fact table where each row represents a single business transaction event. Most common fact type.
31MDXSSASMultidimensional Expressions — query language for SSAS multidimensional cubes.
32DAXPower BI / SSAS TabularData Analysis Expressions — formula language for SSAS Tabular models and Power BI.
33DSVSSASData Source View — a logical view of the DWH schema used by SSAS as the basis for cube/model definition.
34SSASArchitectureSQL Server Analysis Services — Microsoft's OLAP engine for building multidimensional or tabular analytical models.
35Connection ManagerSSISDefines the connection string to a data source in SSIS. Each package uses named connection managers (e.g., OLEDB_OLTP, OLEDB_STG) shared at project level. Wrong connection manager = ETL runs against wrong database with no error.
36Control FlowSSISThe top-level SSIS canvas where tasks (Execute SQL Task, Data Flow Task, Execute Package Task) are arranged and connected by precedence constraints.
37Data Flow Task (DFT)SSISAn SSIS task that moves and transforms data between a source and destination. Contains the pipeline: Source → Transformations → Destination.
38Execute SQL TaskSSISAn SSIS Control Flow task that runs a T-SQL statement against a specified connection manager. Used for TRUNCATE, MERGE, and health check queries.
39Named QuerySSASA SQL SELECT statement saved inside the DSV that acts as a virtual table. Used to apply Is_Current=1 filters on SCD Type 2 dimensions before the cube reads them.
40Attribute RelationshipSSASDefines how dimension attributes relate to each other. Must follow Star pattern (all attributes relate to the Key Attribute directly). Chain relationships cause processing errors and wrong aggregations.
41Measure GroupSSASA logical group of measures inside the SSAS cube, sourced from one fact table. Each fact table in the DWH becomes one measure group in the cube.
42Cube ProcessingSSASThe operation that reads data from the DWH via the DSV and loads it into the SSAS cube's pre-aggregated storage. Must be re-run after every ETL load to reflect new data.
43Live ConnectionPower BIA Power BI connection mode where the report queries the SSAS cube in real time. No data is imported into Power BI — the cube is the single source of truth.
44Row-Level Security (RLS)Power BIA Power BI feature that restricts which rows a user can see in a report based on their identity. Defined as DAX filter rules per role.
45Scheduled RefreshPower BIAn automated process in Power BI Service that re-runs the data load at defined intervals. For Live Connection to SSAS, refresh is controlled by the SSIS ETL schedule — not Power BI.
46Staging LayerArchitectureAn intermediate database that holds an exact, unmodified copy of source data. All transformations happen FROM staging TO the DWH — never from source directly to DWH. Provides a recovery point, auditability, and data quality buffer.
47Degenerate DimensionSchema DesignA dimension attribute stored directly in the fact table with no corresponding dimension table — because it has no descriptive attributes beyond its key (e.g., invoice number, order number, ticket ID). Acts as a drill-down handle.
48Role-Playing DimensionSchema DesignA single dimension used multiple times in the same fact table under different aliases (e.g., Dim_Date used as Order Date, Ship Date, and Due Date). Each role is an alias/view over the same physical dimension table.
49Junk DimensionSchema DesignA dimension table that consolidates multiple low-cardinality flags and codes (e.g., payment method, order status, return flag) into a single table rather than creating many tiny dimension tables. Reduces fact table width.
50Conformed DimensionArchitectureA dimension shared identically across multiple fact tables and subject areas (e.g., Dim_Customer used in both a Sales fact and a Support fact). Enables cross-process analysis and is a core Kimball design principle.
51Natural KeySchema DesignThe business-meaningful identifier from the source OLTP system (e.g., CustomerID, ProductCode). Preserved alongside the surrogate key in DWH dimension tables to enable joins back to the source system.
52Weighting FactorSchema DesignA decimal column in a bridge table (M:M resolution) that allocates a measure across multiple parents. Weights per parent entity must sum to exactly 1.0. Used in revenue attribution and cost allocation scenarios.
53BRT (Business Requirements Table)MethodologyThe 5-axis requirements framework used before any schema design: WHAT (products), WHO (customers), WHERE (geography), WHEN (time), HOW MUCH (measures). One row per business process. Must be signed off before design begins.
⭐ Table B — Star vs Snowflake Decision Framework
Factor⭐ Star Schema❄️ SnowflakeRecommendation
Query PerformanceFaster — fewer joinsSlower — more joins⭐ Star
StorageMore (denormalised)Less (normalised)Depends
MaintenanceSimpler — fewer tablesComplex — many sub-dims⭐ Star
Power BI / SSASRecommended — native supportRequires additional modelling⭐ Star
Data RedundancyHigher (repeated dim values)Lower (values in sub-dims)❄️ Snowflake
Kimball MethodologyDefault recommendationOnly if explicitly justified⭐ Star
ETL ComplexitySimpler flat-dim loadsMore joins to build dims⭐ Star
RecommendationDefault for most DWH projectsOnly when dims > 1M rows or storage critical⭐ Star for most cases
📋 Table C — Table Role Classification Guide
Table RoleCharacteristicsDWH DestinationExamples
Transaction/EventRecords individual business events. Has date, quantity, amount. High row volume. References master tables via FK.Fact TableOrders, Invoices, Payments, Shipments
Master/ReferenceDescribes business entities. Relatively static. Used as lookup by transaction tables. Lower row count.Dimension TableCustomers, Products, Employees, Branches
Bridge/JunctionResolves M:M between two master tables. Usually 2-3 columns. May include weighting/allocation factor.Bridge TableOrderProducts, StudentCourses
History/StatusTracks changes in state over time. Has status column and effective dates. Collapsed into SCD Type 2 dim.Dim (SCD Type 2)CustomerStatusHistory, AccountStateHistory
Lookup/CodeSmall reference tables of codes and descriptions.Dim or Junk DimStatusCodes, CountryCodes, PaymentMethods
Audit/SystemInternal system tables: logging, error tracking, job runs. Not loaded into DWH.Staging/IgnoreETL_Log, SYS_Config, AuditTrail

✅ 4-Package Architecture ChecklistEvery SSIS solution must follow this 4-package architecture
Extract Package built
One DFT per source table, all parallel, OLE DB Source to OLE DB Destination (Staging)
Dimensions Package built
Loads all dimensions in correct dependency order using T-SQL MERGE
Facts Package built
Surrogate key lookups complete, COALESCE to SK=-1 on all FK columns, idempotency guard applied
Master Orchestrator Package built
Chains Extract → Dims → Facts using Execute Package Tasks with Project Reference
